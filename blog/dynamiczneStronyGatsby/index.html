GraphQL - co to jest i jak si tego uywa

GraphQL jest to sposb komunikowania si z serwerem API. Cechuje si tym, e wystpuje tutaj jeden adres pod ktrym s dane, adostp do nich zaley od zapytania, ktre skonstruujemy.
Korzystajc z graphQL dostpny jest dla nas "Plac zabaw", na ktrym moemy tworzy zapytania, a nastpnie je wykorzysta je wkodzie strony.

okno graphql

Narzdzie to jest wyposaone wdokumentacj, ktra zawiera informacje omoliwych zapytaniach, dziki czemu moemy dowolnie ukada zapytania wzalenoci od naszych potrzeb.
Przykadowe zapytanie wyglda nastpujco:

const data = useStaticQuery(graphql`
        query {
            site {
                siteMetadata{
                    title
                }
            }
        }
    `)

Wyej wymienione zapytanie wykorzystywane jest w nagwku strony. Wyszukuje on title strony, o ktrym informacje znajduj si wgatsby-config.js

module.exports = {
  siteMetadata: {
    title: "Serverless Adventure",
  },

Pliki Markdown i ich tworzenie

Spotykamy si znimi gwnie na Githubie, przegldajc najrniejsze repozytoria. Wykorzystywane s take wnajrniejszych dokumentacjach.
Co umoliwia markdown? Pozwala nam on na proste i szybkie stylizowanie tekstu, bez potrzeby tworzenia dodatkowych klas, arkuszy styli itp. Skadnia jest prosta iintuicyjna - nauczenie si jej nie zajmuje wiele czasu. Dziki temu pisanie artykuw przy uyciu markdown'a jest znacznie atwiejsze iszybsze. Nie wymaga on od nas ustawiania styli, zaznaczania tekstu, ktry ma by nagwkiem. Wszystko sprowadza si do odpowiednich kombinacji znakw.

ciga skadni markdown wykonana przez CODE2BITS
Strona jest niestety nie dostpna
link do polskiej wersji

Jak wida na zaczonym obrazku skadnia nie naley do tych skomplikowanych.

Dynamiczne tworzenie stron

Dynamiczne tworzenie stron, czyli przypieszenie tworzenia naszego bloga. Czynno, ktra jest bardzo praktyczna dla osb chccych atwo iprosto mc tworzy artykuy, bez dodatkowego stylizowania - dodawania klas etc.

Proces ten nie naley do skomplikowanych, ale jego poprawne skonfigurowanie naley do tych wymagajcych wikszego kombinowania. Na pocztku pobierane s informacje oplikach, zktrych pobierane bd informacje. Ztych informacji tworzony jest tak zwany slug - jest cig znakw, ktry jest zawarty wlinku strony. Jest to unikalny identyfikator dla konkretnych zasobw.
W przypadku plikw markdown jest on tworzony od nazw plikw. Wwyniku tego wanym jest odpowiednie nazywanie plikw.
Wkolejnym etapie jest proces build'a, podczas ktrego pobierana jest zawarto plikw markdown iumieszczana jest na przez nas wczeniej utworzonym szablonie strony.

Z rzeczy, ktre bd nam potrzebne to:

  • pluginy - gatsby-source-filesystem, gatsby-tranformer-remark, gatsby-plugin-sharp, gatsby-remark-relative-images, gatsby-remark-images
  • dowolne pliki markdown - warto jest mie najpierw sam zarys, eby mc sprawdza itestowa ich dziaanie
  • szablon dla artykuw

I tak naprawd to jest wszystko co jest nam do szczcia potrzebne. Pierwszym krokiem jaki trzeba wykona to skonfigurowa pluginy. Wpliku gatsby-config.js wystarczy umieci ten tego typu kod:

    {
      resolve: 'gatsby-source-filesystem',
      options: {
        name: `src`,
        path: `${__dirname}/src/`,
      }
    },
    'gatsby-plugin-sharp',
    {
      resolve: 'gatsby-transformer-remark',
      options: {
        plugins: [
          'gatsby-remark-relative-images',
          {
            resolve: 'gatsby-remark-images',
            options: {
              //maxWidth jest to maksymalna szeroko zdjcia
              maxWidth: 750,
              linkImagesToOriginal: false
            }
          },
        ]
      }
    },

Jest to konfiguracja potrzebnych nam pluginw.

Adnotacja do dokadniejszych informacji opojedyczych pluginw nastronie ich dokumentacji:

  • gatsby-source-filesystem
  • gatsby-plugin-sharp
  • gatsby-tranformer-remark
  • gatsby-remark-relative-images
  • gatsby-remark-images

Nastpnie potrzebujemy upewni si, e nasze pliki markdown s tak samo wykonane. Chcc mie informacje, wedug ktrych nasze posty powinny by wyszukiwane naley umieci na pocztku pliku tabl:

---
title: "tytu strony"
date: "data artykuu format => YYYY-MM-DD"

---

Informacje w tej tabeli mona wyszuka przy uyciu graphQL - wane jest, eby skadnia wygldaa tak: zmienna: "warto".

Na stronie, na ktrej ma by wywietlana lista artykuw, trzeba zaimportowa: graphql, useStaticQuery z 'gatsby'. Pozwoli to nam na tworzenie zapyta, ktre umoliwi nam wyszukiwanie informacji oplikach markdown. Dziki tym zapytaniom bdziemy mogli stworzy list artykuw, a dziki temu utworzy nawigacj na naszej stronie.

import React from 'react'
import { graphql, useStaticQuery } from gatsby

const Blog = () => {
  
  //statyczne zapytanie graphql

  const data = useStaticQuery(graphql`
    query {
      allMarkdownRemark{
        edges{
          node{
            frontmatter{
              title
              date
            }
          }
        }
      }
    }
  `)

  return (
    //Tworzenie listy artykuw

    <div>
      <ul>
        {
          data.allMarkdownRemark.edges.map((edge) =>(
            <li>
              <article>
                <h2>{edge.node.frontmatter.title}</h2>
                <p>{edge.node.frontmatter.date}</p>
              </article>
            </li>
          ))
        }
      </ul>
    </div>

  )
}

Na tej stronie stworzone zapytanie wyszukuje frontmatter - s to wszystkie informacje, ktre znajduj si wtej tabelce, ktr tworzylimy na pocztku pliku markdown. Dlatego te wanym jest umieszczanie tych samych "zmiennych" wewszystkich plikach markdown.

Fragment ten odpowiada zatworzenie elementw html, ktrych zawartoci bd informacje, ktre zostay wyszukane wzapytaniu, ktre wczniej tworzylimy.

Wprzypadku tej witryny lista artykuw zawiera take wstp doartykuu. Chcc jednak utworzy linki dotych podstron potrzebny nam bdzie szablon strony oraz wczeniej wymieniony slug. Aby mc go stworzy bdziemy potrzebowa utworzy plik gatsby-node.js.
W tym pliku bdziemy musieli utworzy dwie funkcje: jedna odpowiada zatworzenie sluga, adruga zabezporednie tworzenie stron. Zanim to wykonamy potrzebowa jeszcze bdziemy szablonu strony.
Jego stworzenie, nie naley do cikich. Wpierw musimy utworzy zapytanie graphql, ktre bdzie dynamiczne, wic nie bdziemy potrzebowa useStaticQuery. Bdzie ono si wykonywa dla kadego sluga, ktry jest stringiem - ma to na celu uniknicia bdw, gdyby slug by liczb. Nastpnie bdzie wyszukiwa informacje, ktre nale do sluga, ktry dotyczy aktualnej strony.
Pniej umieci wyszukane informacje: tytu i dat, wodpowiednie pola. Tre poza tabel w pliku markdown jest interpretowana jako HTMl, wic naley go umieci w jakim kontenerze przy uyciu dangerouslySetInnerHTML. 

import React from 'react'
import { graphql } from 'gatsby'

export const query = graphql`
  query($slug: String!) {
        markdownRemark(fields: { slug: { eq: $slug } }) {
            frontmatter {
                title
                date
            }
            html
        }
  }
`

const Blog = (props) => {
  return (
    <div>
      <h2>
        {props.data.markdownRemark.frontmatter.title}
      </h2>
      <p>
        {props.data.markdownRemark.frontmatter.title}
      </p>
      <div dangerouslySetInnerHTML={{ __html: props.data.markdownRemark.html}}>
      </div>
    </div>
  )
}

Po utworzeniu szablonu nastpnym krokiem jest tworzenie sluga. Chcc utworzy sluga musimy stworzy zmienn, ktra bdzie wykorzystywa path. Dziki temu bdziemy mogli uy jej wcelu wyszukiwania plikw wnaszych funkcjach. Nastpnie wykorzystujemy API Gatsby.js onCreateNode. Ta funkcja bdzie si wykonywaa za kadym razem gdy bdzie tworzony nowy node (struktura danych) lub bdzie on aktualizowany.

const path = require('path')

module.exports.onCreateNode = ({ node, actions }) => {
    const { createNodeField } = actions

    if (node.internal.type === 'MarkdownRemark') {
        const slug = path.basename(node.fileAbsolutePath, '.md')

        createNodeField({
            node,
            name: 'slug',
            value: slug
        })
    }
}

Powyszy kod bdzie tworzy slug'a tylko dla node'w MarkdownRemark - innymi sowy, tylko dla plikw markdown w katalogach, ktre wczeniej zdeklarowalimy wustawieniach gatsby-source-filesystem.
Zanim je stworzy jednak "ucina" rozszerzenie, dziki czemu ma tylko nazw pliku.

Kolejnym krokiem jest stworzenie funkcji, ktra bdzie tworzy strony. Wykorzystamy do tego createPage.
Chcc j wykorzysta, naley uy kodu, ktry jest poniej.

module.exports.createPages = async ({ graphql, actions }) => {
const { createPage } = actions
const blogTemplate = path.resolve('./src/templates/blogtemp.js')
const res = await graphql(`
    query {
        allMarkdownRemark{
            edges {
                node {
                    fields{
                        slug
                    }
                }
            }
        }
    }
`)
res.data.allMarkdownRemark.edges.forEach((edge)=>{
    createPage({
        component: blogTemplate,
        path: `/blog/${edge.node.fields.slug}`,
        context: {
            slug: edge.node.fields.slug,
        }
    })
})
}

Ten kod pobiera szablon strony, a nastpnie tworzy strony, na bazie informacji, ktre zostay pobrane wzapytaniu graphql.

Aktualizacja strony z artykuami

Gdy mamy ju utworzone strony wkodzie strony zartykuami musimy utworzy odnoniki. 

import React from 'react'
import { Link, graphql, useStaticQuery } from gatsby

const Blog = () => {

  const data = useStaticQuery(graphql`
    query {
      allMarkdownRemark{
        edges{
          node{
            frontmatter{
              title
              date
            }
          }
        }
      }
    }
  `)

  return (
    <div>
      <ul>
        {
          data.allMarkdownRemark.edges.map((edge) =>(
            <li>
            <Link to={`/blog/${edge.node.fields.slug}`}>
              <article>
                <h2>{edge.node.frontmatter.title}</h2>
                <p>{edge.node.frontmatter.date}</p>
              </article>
              </Link>
            </li>
          ))
        }
      </ul>
    </div>

  )
}

Aktualnie jestemy w stanie zajrze na strony pojedyczych artykuw.
